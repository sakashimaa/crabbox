## Day 4: cgroups v2 — All Resource Limits

### Goal

```bash
sudo crabbox run --memory 64M --cpus 0.5 --pids 32 /tmp/crabbox/alpine /bin/sh
```

Container cannot exceed 64MB RAM, half a CPU core, or 32 processes.

### Key concept: cgroups v2 is just files

No special syscalls, no crate needed. You create a directory, write numbers
into files, add a PID. The kernel does the rest.

```bash
# This is literally all cgroups v2 is:
mkdir /sys/fs/cgroup/crabbox-abc123          # create a cgroup
echo 67108864 > .../memory.max               # 64MB memory limit
echo "50000 100000" > .../cpu.max            # 50% of one CPU core
echo 32 > .../pids.max                       # max 32 processes
echo $PID > .../cgroup.procs                 # assign our container to it
```

In Rust this is `std::fs::create_dir()` + `std::fs::write()`. That's it.

### Cgroup v2 files reference

| File             | Controls                           | Example value     |
| ---------------- | ---------------------------------- | ----------------- |
| `memory.max`     | Hard memory limit (bytes)          | `67108864` (64MB) |
| `memory.current` | Current usage (read-only)          | `4521984`         |
| `cpu.max`        | CPU bandwidth: `$QUOTA $PERIOD`    | `50000 100000`    |
| `pids.max`       | Max process count                  | `32`              |
| `cgroup.procs`   | PIDs in this cgroup (write to add) | `12345`           |

**CPU math:** period is always 100000μs (100ms).
`--cpus 0.5` → quota = 0.5 × 100000 = 50000 → write `"50000 100000"`.
`--cpus 2.0` → quota = 200000 → two full cores.

**Memory math:** value in bytes.
`--memory 64M` → 64 × 1024 × 1024 = 67108864.

### Steps

**1. Extend CLI with resource flags**

```
crabbox run [OPTIONS] <rootfs> <command> [args...]

Options:
    --memory <LIMIT>    Memory limit (e.g. 64M, 1G, 512K)
    --cpus <FLOAT>      CPU limit (e.g. 0.5 = 50%, 2.0 = 2 cores)
    --pids <COUNT>      Max number of processes
```

These three flags land as `Option<T>` fields on the existing runtime `ContainerConfig` (from Day 1) — **not** a new struct. Day 5 adds TOML config via a *separate* `TomlConfig` deserialization struct that converts into this same runtime type, so both input paths converge. Keeping a single runtime type means the cgroup code below reads `config.memory_limit` / `config.cpu_limit` / `config.pids_limit` regardless of whether the user passed CLI flags or a TOML file.

**2. Parse human-readable sizes (config.rs)**

```rust
/// "64M" → 67108864, "1G" → 1073741824
fn parse_memory(input: &str) -> Result<u64> {
    let input = input.trim();
    let (num, unit) = input.split_at(input.len() - 1);
    let num: u64 = num.parse()?;
    match unit.to_uppercase().as_str() {
        "K" => Ok(num * 1024),
        "M" => Ok(num * 1024 * 1024),
        "G" => Ok(num * 1024 * 1024 * 1024),
        _ => Err(anyhow!("Unknown unit: {unit}. Use K, M, or G")),
    }
}
```

**3. Create cgroups module (cgroups.rs)**

```rust
use std::fs;
use std::path::{Path, PathBuf};

const CGROUP_ROOT: &str = "/sys/fs/cgroup";

pub struct Cgroup {
    path: PathBuf,
}

impl Cgroup {
    pub fn new(container_id: &str) -> Result<Self> {
        // Create our own parent cgroup at /sys/fs/cgroup/crabbox/. We can't
        // enable controllers on the real root because systemd has tasks in
        // it — cgroups v2 returns EBUSY if any process lives in a cgroup
        // when you try to add a controller. An empty parent cgroup avoids
        // this entirely.
        let parent = Path::new(CGROUP_ROOT).join("crabbox");
        fs::create_dir_all(&parent)
            .context("Failed to create /sys/fs/cgroup/crabbox")?;

        fs::write(
            parent.join("cgroup.subtree_control"),
            "+cpu +memory +pids",
        ).context("Failed to enable cgroup controllers")?;

        let path = parent.join(format!("crabbox-{container_id}"));
        fs::create_dir_all(&path)?;
        Ok(Self { path })
    }

    pub fn set_memory_limit(&self, bytes: u64) -> Result<()> {
        fs::write(self.path.join("memory.max"), bytes.to_string())?;
        Ok(())
    }

    pub fn set_cpu_limit(&self, cpus: f64) -> Result<()> {
        let period: u64 = 100_000;
        let quota = (cpus * period as f64) as u64;
        fs::write(self.path.join("cpu.max"), format!("{quota} {period}"))?;
        Ok(())
    }

    pub fn set_pids_limit(&self, max: u64) -> Result<()> {
        fs::write(self.path.join("pids.max"), max.to_string())?;
        Ok(())
    }

    pub fn add_pid(&self, pid: u32) -> Result<()> {
        fs::write(self.path.join("cgroup.procs"), pid.to_string())?;
        Ok(())
    }

    pub fn memory_usage(&self) -> Result<u64> {
        Ok(fs::read_to_string(self.path.join("memory.current"))?.trim().parse()?)
    }
}

impl Drop for Cgroup {
    fn drop(&mut self) {
        // Move processes out, then remove directory
        if let Ok(procs) = fs::read_to_string(self.path.join("cgroup.procs")) {
            for pid in procs.lines() {
                let _ = fs::write(Path::new(CGROUP_ROOT).join("cgroup.procs"), pid);
            }
        }
        let _ = fs::remove_dir(&self.path);
    }
}
```

**4. Wire into container.rs**

Cgroups are set by the **parent** after fork (needs child PID):

```rust
ForkResult::Parent { child } => {
    let cg = Cgroup::new(&container_id)?;

    if let Some(mem) = config.memory_limit {
        cg.set_memory_limit(mem)?;
    }
    if let Some(cpus) = config.cpu_limit {
        cg.set_cpu_limit(cpus)?;
    }
    if let Some(pids) = config.pids_limit {
        cg.set_pids_limit(pids)?;
    }

    cg.add_pid(child.as_raw() as u32)?;
    waitpid(child, None)?;
    // cg dropped here → cleanup
}
```

**5. Why we don't write to the root cgroup directly**

cgroups v2 won't let you add a controller to a cgroup that has running tasks — EBUSY. On any systemd distro (Arch included), the root cgroup always has tasks, so writing `+cpu +memory +pids` to `/sys/fs/cgroup/cgroup.subtree_control` would always fail.

Step 3 already handles this by creating `/sys/fs/cgroup/crabbox/` as our own empty parent cgroup and enabling controllers there. Container cgroups go under it. No EBUSY because our parent has no tasks.

Symptom if you skip the parent: `memory.max` silently doesn't exist in the container's cgroup directory, and writes to it return ENOENT.

**3. Config structs (config.rs)**

We already have a runtime `ContainerConfig` (Day 1) that holds the validated fields `container::run` consumes. The TOML file uses its **own** set of deserialization structs — separate types, not a collision. A conversion method lifts the TOML shape into the runtime type.

```rust
// TOML file shape — deserialization only, never passed to container::run.
#[derive(Deserialize)]
pub struct TomlConfig {
    pub container: TomlContainer,
    pub limits: Option<TomlLimits>,
}

#[derive(Deserialize)]
pub struct TomlContainer {
    pub rootfs: String,
    pub command: String,
    pub hostname: Option<String>,
}

#[derive(Deserialize)]
pub struct TomlLimits {
    pub memory: Option<String>,
    pub cpus: Option<f64>,
    pub pids: Option<u64>,
}

impl TomlConfig {
    /// Convert the on-disk TOML shape into the runtime ContainerConfig.
    pub fn into_runtime(self, args: Vec<String>) -> Result<ContainerConfig> {
        let memory = self.limits.as_ref()
            .and_then(|l| l.memory.as_deref())
            .map(parse_memory).transpose()?;
        let cpus = self.limits.as_ref().and_then(|l| l.cpus);
        let pids = self.limits.as_ref().and_then(|l| l.pids);

        ContainerConfig::new_with_limits(
            PathBuf::from(self.container.rootfs),
            self.container.command,
            args,
            memory, cpus, pids,
        )
    }
}
```

Flow — two input shapes, one runtime type:

```
CLI flags ────┐
              ├──→ ContainerConfig (runtime) ──→ container::run
TOML file ────┘    (same struct, same validation, same cgroup wiring)
```

This means Day 4's cgroup code (`config.memory_limit`, etc.) works unchanged — it only ever sees `ContainerConfig`, never `TomlConfig`.

CLI supports both:

```bash
sudo crabbox run --memory 64M /tmp/crabbox/alpine /bin/sh   # flags
sudo crabbox run --config container.toml                     # file
```

**4. Add `status` subcommand**

```rust
pub fn status(container_id: &str) -> Result<()> {
    let path = Path::new(CGROUP_ROOT).join(format!("crabbox-{container_id}"));

    if !path.exists() {
        bail!("Container {container_id} not found");
    }

    let mem_usage: u64 = read_cgroup_value(&path, "memory.current")?;
    let mem_limit = fs::read_to_string(path.join("memory.max"))?.trim().to_string();
    let pids_cur = fs::read_to_string(path.join("pids.current"))?.trim().to_string();
    let pids_max = fs::read_to_string(path.join("pids.max"))?.trim().to_string();
    let cpu_max = fs::read_to_string(path.join("cpu.max"))?.trim().to_string();

    println!("Container:  crabbox-{container_id}");
    println!("Memory:     {} / {}", format_bytes(mem_usage), mem_limit);
    println!("PIDs:       {} / {}", pids_cur, pids_max);
    println!("CPU:        {}", cpu_max);
    Ok(())
}
```

pub fn list_containers() -> Result<()> {

## Day 6: Networking Basics — Connecting the Container to the Internet

### Goal

```bash
sudo crabbox run --net bridge /tmp/crabbox/alpine /bin/sh

# Inside container:
ping 8.8.8.8        # works!
apk update          # works! container has internet
ip addr             # has its own IP: 172.16.0.2
```

### Key concept: how container networking works

Right now our container has `CLONE_NEWNET` (if we added it) or shares
the host network. We want something in between: the container gets its
own IP but can still reach the internet. This is exactly how Docker's
bridge networking works.

It's built from three Linux primitives:

**1. veth pair** — a virtual ethernet cable with two ends.
One end goes into the container, the other stays on the host.
Think of it as a network cable connecting two machines.

```
┌─────────────────┐         ┌─────────────────┐
│   Container     │         │      Host        │
│                 │         │                  │
│   eth0 ●────────┼─────────┼──● veth-abc123   │
│  172.16.0.2     │  veth   │                  │
│                 │  pair   │                  │
└─────────────────┘         └─────────────────┘
```

**2. Bridge** — a virtual network switch.
Multiple containers connect to it, and it routes traffic between them.
This is what `docker0` is.

```
┌──────────┐  ┌──────────┐
│ Container│  │ Container│
│    .2    │  │    .3    │
└────┬─────┘  └────┬─────┘
     │veth         │veth
     │             │
┌────┴─────────────┴────┐
│   crabbox0 bridge     │
│   172.16.0.1          │
└───────────┬───────────┘
            │
       ┌────┴────┐
       │  Host   │
       │  eth0   │──→ internet
       └─────────┘
```

**3. NAT (iptables)** — translates container IPs to the host IP
for outbound traffic. Without this, packets from 172.16.0.2 would
be dropped by upstream routers (private IP, not routable).

### Steps

**1. Add network namespace — in the child, not the parent**

`CLONE_NEWNET` behaves differently from `CLONE_NEWPID` in `unshare(2)`:

- `CLONE_NEWPID`: *future* forked children land in a new PID ns. Caller stays put. ✓ safe for the parent.
- `CLONE_NEWNET`: the *caller itself* enters an empty net ns. If the parent calls this, it loses access to the host network — and `setup_bridge()` / `ip link add veth ...` would fail because the parent can no longer see the real device tree.

So we split responsibility: the parent's `unshare_namespaces()` stays as PID|NS|UTS (unchanged from Day 2). The **child** unshares NET itself after fork. The parent — still in the host net ns — reaches into `/proc/<child_pid>/ns/net` to wire up the veth from outside.

```rust
// namespaces.rs — new helper for the child to call
pub fn unshare_network() -> Result<()> {
    unshare(CloneFlags::CLONE_NEWNET)
        .context("failed to unshare network namespace")?;
    Ok(())
}
```

The child calls this first thing after fork — before signalling the parent.

**2. Create networking module (network.rs)**

For day 6, we shell out to `ip` and `iptables` commands.
This is how most container runtimes start — even Docker shells out
to iptables. We can replace with netlink crate later.

```rust
use std::process::Command;

const BRIDGE_NAME: &str = "crabbox0";
const BRIDGE_IP: &str = "172.16.0.1/24";
const SUBNET: &str = "172.16.0.0/24";

/// Create the bridge if it doesn't exist (one-time setup)
pub fn setup_bridge() -> Result<()> {
    // Check if bridge already exists
    let status = Command::new("ip")
        .args(["link", "show", BRIDGE_NAME])
        .status();

    if status.is_ok() && status.unwrap().success() {
        return Ok(()); // already exists
    }

    // Create bridge
    run_cmd("ip", &["link", "add", BRIDGE_NAME, "type", "bridge"])?;
    run_cmd("ip", &["addr", "add", BRIDGE_IP, "dev", BRIDGE_NAME])?;
    run_cmd("ip", &["link", "set", BRIDGE_NAME, "up"])?;

    // Enable IP forwarding
    std::fs::write("/proc/sys/net/ipv4/ip_forward", "1")?;

    // NAT: masquerade outbound traffic from containers
    run_cmd("iptables", &[
        "-t", "nat", "-A", "POSTROUTING",
        "-s", SUBNET, "-j", "MASQUERADE",
    ])?;

    // Allow forwarding to/from bridge
    run_cmd("iptables", &["-A", "FORWARD", "-i", BRIDGE_NAME, "-j", "ACCEPT"])?;
    run_cmd("iptables", &["-A", "FORWARD", "-o", BRIDGE_NAME, "-j", "ACCEPT"])?;

    Ok(())
}

/// Create veth pair and connect container to bridge
pub fn setup_container_net(container_pid: u32, container_ip: &str, container_id: &str) -> Result<()> {
    let veth_host = format!("veth-{}", &container_id[..8.min(container_id.len())]);
    let veth_container = "eth0";

    // Create veth pair
    run_cmd("ip", &[
        "link", "add", &veth_host, "type", "veth",
        "peer", "name", veth_container,
    ])?;

    // Attach host end to bridge
    run_cmd("ip", &["link", "set", &veth_host, "master", BRIDGE_NAME])?;
    run_cmd("ip", &["link", "set", &veth_host, "up"])?;

    // Move container end into the container's network namespace
    run_cmd("ip", &[
        "link", "set", veth_container,
        "netns", &container_pid.to_string(),
    ])?;

    // Configure inside the container's namespace
    let netns = format!("/proc/{container_pid}/ns/net");
    run_nsenter(&netns, &[
        "ip", "addr", "add", &format!("{container_ip}/24"), "dev", veth_container,
    ])?;
    run_nsenter(&netns, &["ip", "link", "set", veth_container, "up"])?;
    run_nsenter(&netns, &["ip", "link", "set", "lo", "up"])?;
    run_nsenter(&netns, &[
        "ip", "route", "add", "default", "via", "172.16.0.1",
    ])?;

    Ok(())
}

fn run_cmd(cmd: &str, args: &[&str]) -> Result<()> {
    let status = Command::new(cmd).args(args).status()
        .context(format!("Failed to run: {cmd} {}", args.join(" ")))?;
    if !status.success() {
        bail!("{cmd} {} failed with {status}", args.join(" "));
    }
    Ok(())
}

fn run_nsenter(netns: &str, args: &[&str]) -> Result<()> {
    let status = Command::new("nsenter")
        .args(["--net", netns])
        .args(args)
        .status()?;
    if !status.success() {
        bail!("nsenter command failed: {}", args.join(" "));
    }
    Ok(())
}
```

**3. IP address allocation**

Simple approach: assign sequential IPs from 172.16.0.2 upward.
Store the last used IP in a file:

```rust
const IP_FILE: &str = "/var/lib/crabbox/last_ip";

pub fn allocate_ip() -> Result<String> {
    let last = fs::read_to_string(IP_FILE).unwrap_or_else(|_| "1".to_string());
    let next: u8 = last.trim().parse::<u8>()? + 1;
    if next > 254 {
        bail!("IP pool exhausted");
    }
    fs::create_dir_all("/var/lib/crabbox")?;
    fs::write(IP_FILE, next.to_string())?;
    Ok(format!("172.16.0.{next}"))
}
```

**4. Wire into container.rs**

Networking setup happens in the parent, after fork, before the child
starts doing real work. This requires a synchronization mechanism
(pipe or signal) so the child waits until networking is ready:

The parent must wait for the child's net ns to *exist* before it can wire the veth into `/proc/<child>/ns/net`; the child must wait for the veth to be *wired* before proceeding. That's two sync points → two pipes.

```rust
// Before fork
let (net_ns_ready_r, net_ns_ready_w) = pipe()?;  // child → parent: "my net ns exists"
let (net_wired_r,    net_wired_w)    = pipe()?;  // parent → child: "veth is wired, go"

match unsafe { fork()? } {
    ForkResult::Parent { child } => {
        drop(net_ns_ready_w);
        drop(net_wired_r);

        // 1. cgroups — needs child PID, safe to do immediately.
        setup_cgroups(&config, child)?;

        // 2. Wait for child to unshare its net ns. Without this, the veth
        //    we create below would end up in the HOST net ns because the
        //    child initially inherits it.
        wait_for_signal(&net_ns_ready_r)?;

        // 3. Set up networking from the host net ns.
        if config.enable_net {
            setup_bridge()?;
            let ip = allocate_ip()?;
            setup_container_net(child.as_raw() as u32, &ip, &container_id)?;
        }

        // 4. Release the child.
        signal_ready(&net_wired_w)?;

        waitpid(child, None)?;
    }
    ForkResult::Child => {
        drop(net_ns_ready_r);
        drop(net_wired_w);

        // 1. Enter our own empty net ns (only `lo`, no connectivity yet).
        namespaces::unshare_network()?;

        // 2. Tell parent: /proc/$$/ns/net is now a real target.
        signal_ready(&net_ns_ready_w)?;

        // 3. Block until veth is wired in from the outside.
        wait_for_signal(&net_wired_r)?;

        // 4. Normal setup — hostname, rootfs, mounts, exec.
        namespaces::set_hostname(&hostname)?;
        filesystem::setup_rootfs(&config.rootfs)?;
        filesystem::mount_proc()?;
        filesystem::mount_tmp()?;
        filesystem::exec_command(&config.command, &config.args)?;
    }
}
```

The dance reads: **child makes the net ns → parent wires it → child proceeds**. Skipping either sync point races the two processes and the veth ends up in the wrong namespace.
