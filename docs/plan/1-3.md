## What we're building

A mini-Docker: a program that runs a process in an isolated environment. After days 1–3: `sudo crabbox run /bin/sh` drops you into a shell that believes it's the only process on a separate machine.

---

## Key concepts (read before coding)

### What is Alpine minirootfs and why do we need it

When you run a container, it needs **its own filesystem** — its own `/bin`, `/etc`, `/lib`, etc. Otherwise `chroot` won't work: the process switches into an empty directory and can't find any programs.

Alpine Linux ships a **minirootfs** — a ~3MB archive containing a minimal working Linux system: busybox (sh, ls, cat, ps, etc.), musl libc, and the apk package manager. This is exactly what Docker downloads when you write `FROM alpine` in a Dockerfile.

You download it once, extract it into a directory, and that directory becomes the "hard drive" of your container.

```bash
# Download and extract rootfs (one-time, by hand, to understand what's happening)
mkdir -p /tmp/crabbox/alpine
cd /tmp/crabbox/alpine
curl -O https://dl-cdn.alpinelinux.org/alpine/v3.21/releases/x86_64/alpine-minirootfs-3.21.3-x86_64.tar.gz
tar xzf alpine-minirootfs-3.21.3-x86_64.tar.gz
ls  # bin  dev  etc  home  lib  media  mnt  opt  proc  root  run  sbin  srv  sys  tmp  usr  var
```

### What chroot / pivot_root does

`chroot("./alpine")` tells the kernel: "for this process, `/` now points to `./alpine`". The process runs `ls /` and sees the contents of `./alpine`, not your real root. It **cannot escape upward** — `./alpine` is its entire world.

`pivot_root` is a more secure version: instead of just changing the perception of `/`, it **swaps the actual root mount**, moves the old root into a subdirectory, which can then be unmounted. Docker uses `pivot_root`, not `chroot`.

We'll start with `chroot` (simpler), then migrate to `pivot_root`.

### What namespaces are

Namespaces are the kernel's way of showing a process **its own version of reality**:

| Namespace | What it isolates                       | Flag for clone() |
| --------- | -------------------------------------- | ---------------- |
| PID       | Process tree (gets its own PID 1)      | `CLONE_NEWPID`   |
| Mount     | Mount points                           | `CLONE_NEWNS`    |
| UTS       | Hostname                               | `CLONE_NEWUTS`   |
| Network   | Network stack (later, not in days 1–3) | `CLONE_NEWNET`   |
| User      | UID/GID mapping (optional)             | `CLONE_NEWUSER`  |

These flags are passed to `clone()` (the syscall for creating a new process) via bitwise OR. The kernel reads the flags and creates the corresponding namespaces. We use `unshare()` — same mechanism, but applied to the current process instead of spawning a new one.

### What the `nix` crate is

Safe Rust wrappers over Linux syscalls. Without it you'd have to write `unsafe { libc::chroot(...) }` with raw pointers. With `nix`:

```rust
use nix::unistd::chroot;
chroot("/tmp/crabbox/alpine")?;  // safe, returns Result
```

## Day 1: Project skeleton + CLI + chroot

### Goal

`sudo crabbox run /bin/sh` → land in a chroot environment with Alpine, see only its filesystem.

### Steps

**1. Initialize the project**

```bash
cargo init crabbox
cd crabbox
```

**2. Dependencies (Cargo.toml)**

```toml
[dependencies]
clap = { version = "4", features = ["derive"] }   # CLI parsing
nix = { version = "0.29", features = ["process", "mount", "sched", "hostname"] }  # Syscalls
anyhow = "1"        # Ergonomic error handling
```

**3. CLI interface (main.rs)**

Parse the `run` command with arguments:

```
crabbox run <rootfs_path> <command> [args...]
```

At this stage `rootfs_path` is the path to the extracted Alpine minirootfs.

**4. Basic chroot (container.rs)**

Day 1 logic in pseudocode:

```
fn run_container(rootfs: &Path, cmd: &str, args: &[String]):
    1. Verify rootfs exists and contains /bin/sh
    2. chroot(rootfs)             — switch the root
    3. chdir("/")                 — move into the new root
    4. exec(cmd, args)            — launch the command
```

With the `nix` crate this is literally:

```rust
use nix::unistd::{chroot, chdir, execvp};

chroot(rootfs_path)?;
chdir("/")?;
execvp(&cmd_cstr, &args_cstr)?;
```

**5. Testing**

```bash
# Prepare rootfs (if you haven't already)
mkdir -p /tmp/crabbox/alpine
cd /tmp/crabbox/alpine
curl -O https://dl-cdn.alpinelinux.org/alpine/v3.21/releases/x86_64/alpine-minirootfs-3.21.3-x86_64.tar.gz
tar xzf alpine-minirootfs-*.tar.gz

# Run
sudo cargo run -- run /tmp/crabbox/alpine /bin/sh

# Inside the container, verify:
ls /                  # → alpine filesystem, not your arch
cat /etc/os-release   # → Alpine Linux
```

### Day 1 result

Working chroot. The process sees a foreign filesystem, but is **not yet isolated**: `ps aux` will show all host processes, hostname is the host's, etc.

## Day 2: Namespaces — real isolation

### Goal

The process sees only itself (PID 1), has its own hostname, its own mount points. This is now a real container.

### Steps

**1. unshare namespaces (namespaces.rs)**

Before chroot, call `unshare()` to create new namespaces:

```rust
use nix::sched::{unshare, CloneFlags};

unshare(
    CloneFlags::CLONE_NEWPID    // own PIDs
    | CloneFlags::CLONE_NEWNS   // own mounts
    | CloneFlags::CLONE_NEWUTS  // own hostname
)?;
```

**Important PID namespace nuance**: `unshare(CLONE_NEWPID)` only affects **child processes**, not the current one. So after `unshare` you need to `fork()`, and continue setup in the child process.

```rust
use nix::unistd::{fork, ForkResult};

match unsafe { fork()? } {
    ForkResult::Parent { child } => {
        // Wait for the container to finish
        waitpid(child, None)?;
    }
    ForkResult::Child => {
        // We're in the new PID namespace, our PID = 1
        setup_filesystem(rootfs)?;
        setup_hostname("crabbox")?;
        exec(cmd, args)?;
    }
}
```

**2. Mount /proc (filesystem.rs)**

Without its own `/proc`, `ps` will show host processes (it reads them from `/proc`). Mount our own:

```rust
use nix::mount::{mount, MsFlags};

// Inside chroot, after pivot
mount(
    Some("proc"),
    "/proc",
    Some("proc"),
    MsFlags::empty(),
    None::<&str>,
)?;
```

**3. Set hostname**

```rust
use nix::unistd::sethostname;
sethostname("crabbox")?;
```

**4. Testing isolation**

```bash
sudo cargo run -- run /tmp/crabbox/alpine /bin/sh

# Inside the container:
hostname         # → crabbox (not your arch hostname)
ps aux           # → only PID 1 (/bin/sh) and PID 2 (ps)
ls /             # → alpine rootfs
mount            # → only our mounts
echo $$          # → 1
```

### Day 2 result

Full isolation: own PIDs, own hostname, own filesystem. The process genuinely believes it's alone on the machine.

## Day 3: pivot_root + cleanup + UX

### Goal

Replace chroot with pivot_root (more secure), add proper error handling, cleanup on exit, container ID generation.

### Steps

**1. Migrate to pivot_root (filesystem.rs)**

`pivot_root(new_root, put_old)` switches the root and moves the old root into `put_old`, which can then be unmounted:

```rust
use nix::unistd::pivot_root;
use nix::mount::{mount, umount2, MntFlags, MsFlags};

fn setup_rootfs(rootfs: &Path) -> Result<()> {
    // 1. Bind mount rootfs onto itself (pivot_root requirement)
    mount(Some(rootfs), rootfs, None::<&str>,
          MsFlags::MS_BIND | MsFlags::MS_REC, None::<&str>)?;

    // 2. Create directory for old root
    let put_old = rootfs.join("oldroot");
    std::fs::create_dir_all(&put_old)?;

    // 3. pivot_root
    pivot_root(rootfs, &put_old)?;

    // 4. Move into the new root
    chdir("/")?;

    // 5. Unmount old root
    umount2("/oldroot", MntFlags::MNT_DETACH)?;
    std::fs::remove_dir("/oldroot")?;

    // 6. Mount /proc, /dev, /tmp
    mount(Some("proc"), "/proc", Some("proc"), MsFlags::empty(), None::<&str>)?;
    mount(Some("tmpfs"), "/tmp", Some("tmpfs"), MsFlags::empty(), None::<&str>)?;

    Ok(())
}
```

**2. Container ID generation**

Short hex ID for identification (like Docker):

```rust
fn generate_id() -> String {
    use std::time::{SystemTime, UNIX_EPOCH};
    let ts = SystemTime::now().duration_since(UNIX_EPOCH).unwrap();
    format!("{:08x}", ts.subsec_nanos())
}
// → "7f3a2b1c"
```

**3. Graceful cleanup**

On container exit — unmount all mount points, remove temporary files. Implement via `Drop` trait or signal handling.

**4. Proper errors**

Replace all `unwrap()` calls with `anyhow::Result` and context:

```rust
chroot(rootfs).context("Failed to chroot into rootfs — does the path exist?")?;
```

### Day 3 testing

```bash
sudo cargo run -- run /tmp/crabbox/alpine /bin/sh

# Inside:
ls /          # no /oldroot (unmounted it)
ps aux        # only our processes
mount         # only /proc and /tmp
hostname      # crabbox-7f3a2b1c

# After exit (Ctrl+D):
# Clean, no leftover mounts or directories
```
